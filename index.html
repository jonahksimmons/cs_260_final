How to fix bugs

And why your questions suck

Setting the stage

The situation is all too familiar. We take on a project too ambitious for our skill level and hit a bug we can not seem to fix. We tear our hair out trying to solve it on our own, but eventually seek guidance from a friend or cry out to Stack Overflow for help. We are either met with nothing, a snarky response, or an answer that completely misses the mark. We conclude that the world hates us and decide to go play Minecraft instead.

No one is perfect; us programmers know this all too well. Handling the inevitable problems in our code is vital not only to your ability to get the task done, but also to learn and grow as a developer. When we don't know the solution ourselves, we ask others for advice. But why do some of our questions go unanswered while others are quickly attended to? The common culprit is the quality of our questions.

What is a "good" question?

There are many ways of approaching this. For this, the goal is to make questions that people will answer easily. Aside from readability and other general attributes, programming questions require some important pieces of information if you want solutions. From years of generating errors and fixing the errors of others, these are the most important guidelines for writing technical questions.

Give background

This is often the most overlooked part. Every project is set up differently and is made up of many moving parts. What language is it written in? What framework or libraries does it use? What platforms are you targeting?

I'm writing a bash script to search through a file for multiple strings, but it always shows nothing. Running it produces an error code of 1. This is the command I'm using:

cat users.txt | grep "Name" | grep "Age"

Say what you want it to do

Knowing the error messages is helpful, but knowing what you want to do is even more helpful. The issue you are having could be your syntax, but it could also be your approach. What purpose do you want this part of your program to solve? How do you want it to fit into the rest of your project?

I'm writing a bash script to search through a file for multiple strings, but it always shows nothing. Running it produces an error code of 1. This is the command I'm using:

cat users.txt | grep "Name" | grep "Age"

Explain what it actually does

Now list the problem. This could include error messages, command line output, or anything else that the program interacts with.

I'm writing a bash script to search through a file for multiple strings, but it always shows nothing. Running it produces an error code of 1. This is the command I'm using:

cat users.txt | grep "Name" | grep "Age"

More info is better than less info

If you have error messages, show the full message. If you are working on a front end, show screenshots. This can save a lot of back and forth with people asking for logs or stack traces.

I'm writing a bash script to search through a file for multiple strings, but it always shows nothing. Running it produces an error code of 1. This is the command I'm using:

cat users.txt | grep "Name" | grep "Age"

Show relevant code

Although it might be tempting, don't just skip to this step. It is really important, but harder to decipher without the previous information.

I'm writing a bash script to search through a file for multiple strings, but it always shows nothing. Running it produces and error code of 1. This is the command I'm using:

cat users.txt | grep "Name" | grep "Age"

With this information, someone can now propose a solution. In this case, the correct code would be

grep -e Name -e Age users.txt

Asking questions is really important for learning and asking good questions help speed up that growth. Hopefully this guide will aid you when you next cry into the cold, unforgiving void of Stack Overflow or bug a senior dev for the 100th time. And maybe, just maybe, you will get a answer to match the high quality of your question.
